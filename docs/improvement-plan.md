## 1+2. 제보 시스템 (버튼 + 관리자 게시판)

두 요구는 하나의 기능(제보 파이프라인)의 입구와 출구이므로 함께 구현한다.

### 사용자 흐름
```
[제보하기 버튼] → 제보 모달(유형 선택 + 내용) → 서버 저장(Supabase)
                                                      ↓
                          관리자 #admin 로그인 → [제보 게시판] 탭에서 열람/처리
```

### UI
- **제보 진입점 2곳**
  - 상단바: "📢 제보하기" 버튼 (신규 물놀이장 제보용)
  - 각 장소 카드/정보창: "정보 수정 제보" 링크 (해당 장소 ID 자동 첨부)
- **제보 모달** (기존 리뷰 모달 패턴 재사용)
  - 유형 select: `새 물놀이장` / `정보 수정(기간·시간 등)` / `편의시설 정보` / `기타`
  - 내용 textarea (필수, 500자)
  - 연락처 (선택 — 회신용)
- **관리자 게시판** (기존 admin 모달 확장)
  - 로그인 성공 시 "제보 게시판" 목록 노출: 유형 배지 · 대상 장소 · 내용 · 날짜
  - 행별 액션: `처리완료 토글` / `삭제`
  - 상태 필터: 신규 / 완료

### 데이터 모델 (Supabase `reports` 테이블)
```sql
create table public.reports (
  id         uuid primary key default gen_random_uuid(),
  place_id   text,                          -- null이면 신규 장소 제보
  type       text not null,                 -- new-place | correction | amenity | etc
  content    text not null,
  contact    text,
  status     text not null default 'new',   -- new | done
  created_at timestamptz not null default now()
);
alter table public.reports enable row level security;
-- 정책 없음: 서버(secret 키)만 접근. anon으로는 읽기/쓰기 모두 불가.
```
> 리뷰 테이블과 달리 **anon 정책을 아예 만들지 않는다.** 제보는 서버 경유만 허용 → 관리자 외 열람 원천 차단(요구 2 충족).

### API (server.js 추가)
| 메서드 | 경로 | 인증 | 비고 |
|---|---|---|---|
| POST | `/api/reports` | 없음 | 리뷰와 동일한 rate limit 재사용(10분 3건) |
| GET | `/api/admin/reports?status=` | X-Admin-Token | 게시판 목록 |
| PATCH | `/api/admin/reports/:id` | X-Admin-Token | status 변경 |
| DELETE | `/api/admin/reports/:id` | X-Admin-Token | 삭제 |

### 규모
- 서버: 엔드포인트 4개 + reviewStore 패턴 복제 → **소~중**
- 프론트: 모달 1개 + admin 모달 확장 → **중**
- 사진 첨부는 v1 제외 (텍스트만). 필요해지면 리뷰의 base64 방식 재사용.

## 3. 편의시설 픽토그램 (기본 4종)

### 데이터 모델
`places`에 `amenities` 필드 추가 (JSON). **3상태**: `true`(있음) / `false`(없음) / 없음·`null`(미확인).

```json
"amenities": { "toilet": true, "shower": false, "changing": null, "store": true }
```
- `data/places.json` + Supabase `places` 테이블에 `amenities jsonb` 컬럼 추가
- 구버전 프론트(pplx.app)는 모르는 필드를 무시하므로 **데이터 먼저 넣어도 무해**

### 표시 방식
- **카드(목록)**: 확인된 "있음" 아이콘만 한 줄로 압축 표시 → `🚻 🚿 🏪`
- **정보창/상세**: 전체 항목 3상태 표시 — 있음(컬러) · 없음(회색+취소) · 미확인(반투명 `?`) + "시설 정보 제보" 링크
- 이모지 사용 (앱이 이미 🌊 〰️ 등 이모지 스타일, 별도 아이콘 폰트 불필요)

| 항목 | 아이콘 | 키 |
|---|---|---|
| 화장실 | 🚻 | toilet |
| 샤워실 | 🚿 | shower |
| 탈의실 | 👕 | changing |
| 편의점/매점 | 🏪 | store |

### 데이터 수집 전략 (212곳을 어떻게 채우나)
1. **마포구 11곳 수동 조사로 시작** — 기존 "마포 우선" 전략과 동일. 관리자가 직접 확인해 시드.
2. **Kakao Local 카테고리 API 자동 스캔** — 좌표 반경 300m 내 편의점(`CS2`)·주차장(`PK6`) 존재 여부를 스크립트로 일괄 판정 (`scripts/scan-amenities.js`, 보유한 REST 키 사용). 편의점·주차는 자동화 가능.
3. **화장실·샤워·탈의는 자동 판별 불가** → **제보 시스템(1번)과 연동**: 제보 유형 `편의시설 정보`로 방문자가 알려주면 관리자가 확인 후 반영. 제보→검수→데이터 갱신 루프.

## 4. 추가 정보 항목 제안 (픽토그램 확장)

아이 동반 방문객이 실제로 가장 많이 묻는 것 기준으로 선별:

| 항목 | 아이콘 | 키 | 왜 필요한가 | 수집 방법 |
|---|---|---|---|---|
| 주차 | 🅿️ | parking | 차로 이동하는 가족 최다 질문 | Kakao API 자동(반경 내 주차장) |
| 그늘막/쉼터 | ⛱️ | shade | 폭염 속 보호자 대기 공간 | 수동/제보 |
| 유아 적합(얕은 수심) | 👶 | toddler | 대상 연령 판단 | 수동/제보 |
| 안전요원 상주 | 🦺 | lifeguard | 안전 최우선 관심사 | 구청 공지/제보 |
| 수유실·기저귀교환대 | 🍼 | nursing | 영유아 동반 필수 정보 | 수동/제보 |
| 유모차 접근 | ♿ | stroller | 접근성 | 수동/제보 |

- **권장 구성: 기본 4 + 확장 6 = 총 10종**, 카드에는 "있음"만 압축 표시하므로 아이콘 과밀 없음.
- 상세 화면에서만 10종 전체 노출.
- v1에서는 스키마에 10종 키를 모두 정의하되, 데이터는 채워지는 대로 표시 (미확인은 `?`).

## 5. 서울 자치구 필터 개선

문제: 서울 탭 선택 시 칩 26개(전체+25구) — 두 줄 이상 감싸며 스캔이 어려움. 경기(7개)는 현행 유지.

### 검토안

| 안 | 내용 | 장점 | 단점 |
|---|---|---|---|
| **A. 권역 그룹 (권장)** | 서울시 공식 5권역(도심·동북·서북·서남·동남) 서브탭 → 선택 시 소속 구만 칩 표시 | 최대 칩 수 8개로 감소, 기존 탭 패턴 재사용, diff 최소 | 클릭 1회 증가 |
| B. 현 지도영역 필터 | "이 지역에서 검색" 버튼 (부동산 앱 방식) | 지도 중심 UX와 자연스러움 | 구현량 큼, 칩 문제 자체는 남음 |
| C. 내 주변 | 위치 권한 받아 가까운 순 정렬 | 방문 목적과 정확히 일치 | HTTPS 필요, 권한 거부 시 무용 |
| D. 검색 자동완성 | 구 이름 입력창 | 칩 제거 가능 | 모바일 타이핑 부담, 탐색성 저하 |

### 권장안 A 상세
서울시 공식 5권역 분류:
```
도심권(3): 종로 중구 용산
동북권(8): 성동 광진 동대문 중랑 성북 강북 도봉 노원
서북권(3): 은평 서대문 마포        ← 마포구 기본값 → 서북권 자동 선택
서남권(7): 양천 강서 구로 금천 영등포 동작 관악
동남권(4): 서초 강남 송파 강동
```
- 서울 탭 선택 시에만 권역 칩 줄 노출: `전체 · 도심 · 동북 · 서북 · 서남 · 동남`
- 경기 탭은 현행 그대로(7칩)
- 구현: `app.js`에 권역 상수 + `regions()` 필터 한 단계 추가, 기존 탭 CSS 재사용 → **소**
- B·C안은 후속 개선으로 보류 (특히 C "내 주변"은 모바일 사용자에게 가치가 커서 다음 라운드 후보)
